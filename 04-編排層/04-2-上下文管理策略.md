# 04-2 上下文管理策略
> 主動管理 Claude Code 的 context 窗口，讓每一個 token 都用在真正重要的資訊上。

## 學習目標
- 理解 context 窗口的填充速度與性能下降的關係
- 識別 context 過載的 4 個早期症狀，在品質下降前介入
- 掌握 8 個具體管理技巧，尤其是 Document & Clear 法
- 在複雜多階段任務中，設計出不爆 context 的工作流

---

## 正文

### 1. 為什麼上下文管理是最重要的技能

Claude Code 擁有 200K token 的 context 窗口，這個數字看起來很充裕，但在實際開發工作中，填充速度遠比直覺預期的快。

一個中等規模的任務——例如為 `src/reviewer.ts` 新增一個功能，涉及讀取 10 個相關檔案、5 次工具調用輸出、3 輪根據回饋的修正——可以輕易消耗 50,000 到 80,000 tokens。如果這個任務還包含調查一個 bug，加上執行測試的輸出，突破 100K 並不罕見。

**性能與使用量的關係不是線性的**。Context 使用率超過 40% 後，回應品質開始出現可察覺的下滑；超過 80% 後，Claude 處理複雜推理的能力明顯受限；到達 95% 時，系統觸發自動壓縮，但壓縮是有損的——它保留結構，丟棄細節。

核心原則：自動壓縮是安全網，不是策略。等到被動觸發壓縮，你已經讓 Claude 在降級狀態中工作了一段時間。主動管理 context 比被動等待壓縮，在工作品質和 token 效率上都更優。

### 2. Context 過載的 4 個早期症狀

識別過載比等待系統觸發更重要。以下 4 個症狀是可觀察的警示信號：

**症狀 1：指令遺忘**

你在 session 開頭告知「所有函數必須加 JSDoc 註解」，但 10 輪對話後，Claude 開始產出沒有 JSDoc 的函數。這不是 Claude 的錯，是 context 中的早期指令已被後來的工具調用輸出稀釋。

**症狀 2：已修正的錯誤重現**

你在第 5 輪修正了一個命名慣例問題，但第 15 輪 Claude 又回到原本的錯誤命名。這是 context 中關鍵修正記錄已被推出有效範圍的跡象。

**症狀 3：回應變短且細節減少**

Claude 的回應從詳細的實作說明，變成越來越簡短的概要。這是 Claude 在有限 context 下進行取捨的結果——它優先保留核心推理，犧牲細節輸出。

**症狀 4：相同 prompt 成功率下降**

一個在 session 初期每次都能正確執行的指令，在 session 後期開始不穩定，有時成功有時失敗。這是 context 中背景資訊不完整導致的不一致行為。

出現任何一個症狀，就應該立即採取管理措施，而不是繼續嘗試同一個 prompt。

### 3. 8 個管理技巧

| 技巧 | 做什麼 | 何時使用 | 節省效果 |
|---|---|---|---|
| `/clear` | 完全重置 context，從零開始新 session | 切換到不相關的任務，或完成一個完整階段後 | 100%（完全重置） |
| `/compact [指示]` | 壓縮 context 歷史，保留摘要 | 接近窗口上限但任務未完成，不能 /clear | 壓縮後節省 50-70% |
| Subagent 隔離 | 把複雜調查委派給 Subagent 執行 | 任何需要大量中間步驟的研究或除錯任務 | 主對話完全不膨脹 |
| `/btw` | 旁問模式，回答不進入對話歷史 | 需要快速釐清一個小問題，但不想增加 context | 不增加 context |
| Plan Mode | 以唯讀方式探索與分析，不執行任何修改 | 理解需求、評估方案、規劃架構的階段 | 節省 40-60% token |
| Document & Clear | 把計畫寫入 .md 檔 → /clear → 引用繼續 | 複雜多階段功能開發的標準工作流 | 每階段從乾淨 context 開始 |
| 短 session | 主動在 30-45 分鐘後結束 session 並重啟 | 日常開發的時間節奏管理 | 防止 context 無聲累積 |
| CLAUDE.md 精簡 | 控制 CLAUDE.md 在 200 行以內，只保留必要指令 | 永遠適用，在首次建立後定期審查 | 減少每次 session 載入消耗 |

### 4. Document & Clear 法：最重要的一個技巧

Document & Clear 是高產出工程師在使用 Claude Code 處理複雜功能時最常見的工作流。這個方法最早由 Boris Cherny 在 Anthropic 內部推廣，後來被多位工程師採用並驗證。

**四個步驟**：

**步驟 A：讓 Claude 完成一個階段的工作**

定義清楚一個「階段」的邊界。例如：「分析現有架構並規劃新功能的實作方案」是一個階段；「根據規劃實作功能」是下一個階段。不要讓一個 session 承擔整個端到端的工作。

**步驟 B：讓 Claude 把計畫、進度、關鍵決策寫入 .md 檔案**

在 session 結束前，要求 Claude 把這一階段產出的所有重要資訊整理成文件。這個文件應該包含：決定了什麼（含決策理由）、發現了什麼（含關鍵數據）、下一步要做什麼（含具體指令）。文件要自給自足——一個對這個 session 沒有任何記憶的 Claude，讀完這個文件後應該能無縫繼續工作。

**步驟 C：`/clear` 重置上下文**

乾淨地結束這個 session。所有的中間產物、調查過程、試錯記錄一起清空。

**步驟 D：在新的乾淨 context 中，用 `@filename.md` 引用文件繼續工作**

新 session 開頭直接引用文件：「根據 @docs/plan-analyzer.md 的計畫，繼續實作第二階段」。Claude 在完整的 context 空間中，以高品質完成後續工作。

**為什麼這比讓 context 累積更好**：每個階段在乾淨的 context 中執行，Claude 的推理品質維持在高水位；文件的存在讓整個開發過程可追蹤、可重現；萬一需要回頭修正，文件記錄了每個決策的理由。

### 5. 自動壓縮的行為與邊界

了解自動壓縮的機制，才能知道為什麼不應該依賴它。

**觸發條件**：context 使用量達到 200K 窗口的 95%。

**壓縮行為**：系統保留 `CLAUDE.md` 的內容、最近幾輪對話的關鍵訊息、以及對話的摘要結構。其餘的工具調用詳細輸出、中間推理步驟、早期的具體指令，均被壓縮或丟棄。

**壓縮的代價**：壓縮後，Claude 對早期的決策、指令、和上下文的掌握變得模糊。如果你在 session 初期建立了某個重要的理解（例如「這個函數的邊界情況是 X」），壓縮後 Claude 可能不再記得。

**正確的定位**：自動壓縮是防止 session 完全崩潰的安全網。在 context 用盡前，它讓你還能繼續對話。但它不是讓你在 80% context 使用率下繼續工作的理由——那個使用率已經超過了品質下滑的臨界點。

## 實作範例：AI Dev Assistant 大功能開發的上下文管理
> 主案例延續：在 AI Dev Assistant 專案中，用 Document & Clear 法開發 analyzeFile 功能，全程保持 context 低使用量。

**目標**：為 `src/reviewer.ts` 新增 `analyzeFile` 函數，示範如何在不爆 context 的情況下完成一個需要分析現有程式碼、規劃、實作、測試的完整功能。

**前置條件**：已完成 04-1（自訂 Subagents），ai-dev-assistant 專案存在 `src/reviewer.ts` 和對應的測試檔案。

**步驟**：

**步驟 1**：啟動 Claude Code 並切換到 Plan Mode

```bash
claude
```

進入互動模式後，按 `Shift+Tab` 兩次，切換到 Plan Mode（底部狀態欄顯示 Plan Mode）。

**步驟 2**：在 Plan Mode 中進行需求分析

輸入：

```
分析 AI Dev Assistant 的現有結構，理解 src/reviewer.ts 的架構，
規劃如何加入一個 analyzeFile(filePath: string): Promise<ReviewResult> 函數。
包含：函數簽名、與現有函數的整合點、需要新增的型別定義、測試案例清單。
```

Plan Mode 的好處：Claude 閱讀並分析所有相關檔案，但不執行任何修改操作。這個階段只消耗分析所需的 token，不會有工具調用輸出塞滿 context。

**步驟 3**：讓 Claude 把計畫寫入文件

在 Plan Mode 分析完成後，切回 Normal Mode（按 `Shift+Tab` 一次），然後輸入：

```
把這個分析結果和實作計畫寫入 docs/plan-analyzeFile.md。
文件必須包含：現有架構摘要、analyzeFile 的完整函數簽名、整合點說明、
需要新增的型別、測試案例清單（至少 5 個），以及實作時需要注意的邊界情況。
```

等待 Claude 建立 `docs/plan-analyzeFile.md`。

**步驟 4**：重置上下文

```
/clear
```

這個 session 產生的所有中間產物、分析過程、Plan Mode 的讀取記錄，全部清空。此時 `docs/plan-analyzeFile.md` 已儲存在磁碟上，不受 `/clear` 影響。

**步驟 5**：在乾淨 context 中開始實作

```
根據 @docs/plan-analyzeFile.md 的計畫，在 src/reviewer.ts 中實作
analyzeFile 函數。先實作函數本體，再新增對應的 Jest 測試。
```

Claude 讀取 `docs/plan-analyzeFile.md`，在完整的 200K context 空間中，以高品質執行實作。

**步驟 6**：執行測試確認結果

在 Claude 完成實作後，確認測試通過：

```
執行 npm test -- --testPathPattern="reviewer" 並回報結果
```

**步驟 7**：遇到測試失敗時，委派 Subagent 調查

如果有測試失敗，不要在主對話中展開調查（這會消耗 context）。改用：

```
@debugger 調查 src/reviewer.test.ts 中 analyzeFile 相關的失敗測試，修復並確認所有測試通過
```

debugger Agent 在獨立 context 中完成調查與修復，主對話只收到「根因、修復內容、測試通過」的摘要。

**步驟 8**：記錄完成狀態（為下一個階段做準備）

功能完成後，讓 Claude 更新文件：

```
更新 docs/plan-analyzeFile.md，在末尾加上「實作完成」章節，
記錄：最終的函數簽名（如有調整）、測試通過率、以及發現的任何與原計畫不同之處。
```

**預期結果**：
- `analyzeFile` 函數實作完成，所有 Jest 測試通過
- 主對話在整個過程中的 context 使用量維持低位：Plan Mode 分析、`/clear` 重置、Subagent 隔離三個機制共同作用
- `docs/plan-analyzeFile.md` 作為完整的開發記錄保留在專案中，可供日後參考或作為 PR 說明的基礎

**常見錯誤**：

**錯誤 1：`/clear` 後忘記引用計畫文件，從零重新解釋需求**

這是最常見的失誤。`/clear` 後直接說「繼續實作 analyzeFile」，Claude 不知道 analyzeFile 是什麼，需要重新解釋所有背景，浪費 token 又容易遺漏細節。

正確做法：養成 `/clear` 前先確認文件已存在、`/clear` 後第一句話就用 `@filename.md` 引用的習慣。可以用一句話確認：「在 /clear 前，文件已寫入 docs/plan-analyzeFile.md 了嗎？」

**錯誤 2：在 Plan Mode 中嘗試要求 Claude 編輯檔案，操作被拒絕**

Plan Mode 是唯讀模式，Claude 只能讀取和分析，任何修改操作（Edit、Write、Bash 執行指令）都會被拒絕。

正確做法：Plan Mode 只用於分析和規劃。確認計畫後，切回 Normal Mode（`Shift+Tab` 一次）再請 Claude 把計畫寫入文件，然後再進行實作。記住切換順序：Plan Mode 分析 → Normal Mode 寫文件 → `/clear` → Normal Mode 實作。

## 關鍵重點
- Context 性能下降是非線性的：40% 後開始下滑，80% 後明顯受限，不要等到 95% 才處理
- 4 個早期症狀（指令遺忘、錯誤重現、回應變短、成功率下降）是比系統提示更早的警示信號
- Document & Clear 是最高效的策略：讓每個開發階段都在乾淨的 context 中進行，文件承接跨 session 的連續性
- 自動壓縮是安全網，不是工作策略：依賴它意味著你已經讓 Claude 在降級狀態中工作了
- Subagent 隔離和 Plan Mode 是主動管理 context 的兩個最有力工具，兩者可以組合使用
