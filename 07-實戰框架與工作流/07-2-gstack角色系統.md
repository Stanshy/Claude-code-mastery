# 07-2 gstack 角色系統
> Y Combinator 總裁 Garry Tan 如何用 18 個 Skill 模擬完整工程團隊。

## 學習目標
- 理解 gstack 為什麼用「角色分工」而非「單一巨型 CLAUDE.md」來組織工作流
- 掌握 `/office-hours` 和 `/investigate` 兩個最核心機制的設計邏輯
- 學會從 gstack 提取可直接移植到自己系統的三個做法
- 了解 gstack 的局限，判斷哪些場景適合採用這套系統

## 正文

### 1. 背景：一個 YC 總裁的工程實驗

Garry Tan 是 Y Combinator 現任總裁。他開源了 gstack（github.com/garrytan/gstack），並在 60 天內用這套系統交付了 60 萬行生產代碼。這個數字本身已經值得注意，但更值得注意的是他選擇了什麼樣的設計哲學。

gstack 不是一個堆滿規則的巨大設定檔，而是一個模擬工程團隊組織結構的系統。它的核心假設是：**不同類型的問題需要不同的思維框架，而這些框架不應該在同一個 context window 中互相干擾。**

---

### 2. 設計理念：角色分工，而非規則堆疊

大多數人建立 CLAUDE.md 的方式是不斷添加規則，直到文件變成一個長達幾百行的清單。這個方式有一個根本問題：當你要 Claude 寫程式碼時，關於「如何做 security audit」的規則也在 context 裡佔用空間和注意力。

gstack 的解法是：把不同類型的工作拆成獨立的 Skill 檔案，每個 Skill 模擬一個工程團隊角色。當你需要某個角色的判斷時，才載入那個角色的 context。

gstack 的 18 個 Skill 涵蓋完整工程週期：

| 角色 | 功能 | 對應 Skill |
|------|------|-----------|
| CEO | 重新思考產品方向 | `/rethink` |
| Eng Manager | 鎖定架構決策 | `/lock` |
| Designer | 偵測 AI slop（低品質 AI 輸出） | `/design-review` |
| Reviewer | 找生產 bug | `/review` |
| QA Lead | 瀏覽器實測 | `/qa` |
| Security Officer | OWASP + STRIDE 審計 | `/security` |
| Release Engineer | PR 發布 | `/ship` |

這七個角色涵蓋了從產品決策到部署的整個流程。每個角色都有明確的職責邊界，不會互相滲透。

---

### 3. 核心工作流：Think → Plan → Build → Review → Test → Ship → Reflect

gstack 強制完整循環，不允許跳步。這個循環的設計邏輯是：**每一步的輸出都是下一步的輸入，任何跳過的步驟都會在後面的步驟中以更高的成本出現。**

- **Think**：釐清問題是什麼，為什麼要解決它
- **Plan**：決定如何解決，拆解步驟
- **Build**：實作
- **Review**：從多個角度檢視實作品質（功能、安全、效能、可維護性）
- **Test**：驗證功能正確性
- **Ship**：發布，記錄決策
- **Reflect**：提取本次工作的學習，更新系統

---

### 4. 兩個最值得深入理解的機制

#### /office-hours：在動手前強制回答六個問題

**為什麼這個機制存在：** 大多數工程決策的問題不在於「怎麼做」，而在於「做什麼」和「為什麼做」沒有被充分思考。直接開始寫程式碼是一種衝動，它讓人感覺在進展，但常常是在快速走向錯誤方向。

`/office-hours` 強制你在開始任何大任務前，回答六個問題。這六個問題的設計目標是逼出隱藏的假設和遺漏的考量：

1. 這個功能解決的具體用戶問題是什麼？
2. 成功的定義是什麼？如何驗證？
3. 有哪些技術限制或依賴？
4. 最可能失敗的地方在哪裡？
5. 有沒有更簡單的解法？
6. 這個決策是否可逆？如果錯了，代價是什麼？

在你回答完這六個問題之前，gstack 不允許進入 Build 階段。這個強制暫停的設計，本質上是把「思考成本」前置，而不是讓它在實作過程中以更高的「修正成本」出現。

#### /investigate：修 bug 前必須先找根因

**為什麼這個機制存在：** 修 bug 最常見的錯誤模式是「看到症狀就動手改」。這個做法有兩個問題：第一，你修的可能是症狀而不是原因，問題會以不同形式重現；第二，在沒有理解根因的情況下修改，你不知道這個修改會不會引入新問題。

`/investigate` 強制在任何 bug fix 之前執行根因分析。流程是：

1. 重現問題（確認問題是真實存在的）
2. 隔離變數（確認問題的邊界）
3. 形成假設（為什麼這個問題會發生）
4. 驗證假設（用最小的方式確認假設正確）
5. 提出修正方案（修正根因，而不是症狀）

只有完成前四步之後，才進入實際修改程式碼的階段。

### 5. 可以直接移植的三個做法

你不需要採用整個 gstack 系統，但這三個做法可以獨立使用，每一個都能立即帶來具體效益：

**做法一：逼問模式**

在開始任何預計超過 4 小時的任務之前，強制回答 5 至 6 個問題。你可以用 gstack 的六個問題作為起點，或根據你自己的領域調整。重點是「在動手前把隱藏假設挖出來」。

**做法二：先找根因**

建立一個個人規則：任何 bug fix，必須先在紙上或 CLAUDE.md 中寫下「根本原因是」，然後再動手修。這個文字化的動作強制你從「感覺」切換到「分析」。

**做法三：完整循環**

不是只做 Build，而是走完 Think → Plan → Build → Review → Test → Ship → Reflect 的完整週期。哪怕每個步驟只花 5 分鐘，也比跳步直接 Build 更可靠。Reflect 步驟尤其值得重視——它是讓你從每次工作中提取可重用知識的唯一機會。

---

### 6. gstack 的局限

採用一套系統之前，先了解它的邊界：

**學習曲線：** 18 個 Skill 需要時間熟悉。在你還不清楚哪個 Skill 對應哪種情境時，切換角色的摩擦會降低效率，而不是提升效率。

**規模假設：** gstack 的設計假設你在處理有一定規模的專案——有真實的安全考量、有需要跨部門協調的架構決策、有需要正式化的發布流程。如果你在做一個週末原型，`/security`（OWASP + STRIDE 審計）可能是過度設計。

**需要客製化：** gstack 是 Garry Tan 為他自己的工作流設計的。他的產品決策週期、技術棧、團隊結構都反映在這 18 個角色的設計中。你需要根據自己的情況調整，而不是直接複製。

## 教學舉例

### 場景：用 /office-hours 避免一個代價高昂的錯誤技術決策

假設你是一個獨立開發者，正在為一個 B2B SaaS 產品開發「報表匯出」功能。你的第一直覺是：「這很簡單，用 `exceljs` 產生 Excel 檔案，讓用戶下載就好。」你準備直接開一個 Claude Code session，給出任務描述，讓它開始寫程式碼。

但你記得 gstack 的 `/office-hours` 機制。在啟動 session 之前，你先強制自己回答六個問題。

**問題一：這個功能解決的具體用戶問題是什麼？**

你的回答：「用戶需要把平台上的數據帶出去，在 Excel 裡做進一步分析。」

這個回答看起來合理，但你繼續往下。

**問題二：成功的定義是什麼？如何驗證？**

你開始思考：成功是「用戶能下載 Excel」？還是「用戶能在 Excel 裡正確分析數據」？如果某個報表有 50 萬筆資料，用戶下載了一個 300MB 的 Excel 檔案，算成功嗎？

你意識到你沒有想清楚數據量的問題。

**問題三：有哪些技術限制或依賴？**

你查了一下：你的 API server 是無狀態的，跑在記憶體 512MB 的容器上。如果某個大客戶有 100 萬筆資料，在記憶體中生成 Excel 會讓容器 OOM 崩潰。你之前沒有想到這個。

**問題四：最可能失敗的地方在哪裡？**

現在很清楚了：大量數據的記憶體問題是最可能的失敗點。其次是「Excel 格式」是否真的是用戶需要的，還是他們其實只需要 CSV，因為他們最終會把 Excel 轉成其他格式。

**問題五：有沒有更簡單的解法？**

你想了想：如果用戶真正的需求是「帶走數據做分析」，那 CSV streaming 可能是更輕量的解法。Server 不需要在記憶體中建構整個檔案，而是用串流方式逐行輸出，記憶體佔用幾乎為零。

**問題六：這個決策是否可逆？如果錯了，代價是什麼？**

如果你先做了 Excel 方案，之後發現大數據量有問題，你需要重寫整個匯出邏輯。代價是高的。如果你先做 CSV streaming，之後要加 Excel 支援，可以在此基礎上擴展，代價是低的。

**結論**

你花了 15 分鐘回答六個問題，得出了一個和原始直覺完全不同的結論：先用 CSV streaming 實作，加上一個明確的記憶體上限（例如最多回傳 10 萬筆，超過提示用戶分批匯出），之後再根據用戶需求評估是否需要 Excel 格式。

這個決策改變了你後來給 Claude 的任務描述：

原始版本：「用 exceljs 實作報表匯出，讓用戶下載 Excel 檔案。」

更新版本：「實作 CSV streaming 匯出，使用 Node.js 的 `stream.Transform`，逐行從資料庫查詢並寫入 response stream。每次查詢最多 1000 筆，總上限 10 萬筆。超過上限時回傳 HTTP 413 並附上錯誤訊息。請先寫一個 unit test 測試 streaming 行為，再實作主體。」

Claude 拿到這個更精確的任務描述後，產出的程式碼品質更高，因為任務的邊界和驗收條件都已經被清楚定義。而這一切，只需要 15 分鐘的逼問思考。

這就是 `/office-hours` 的核心價值：不是讓你花更多時間思考，而是讓你把思考放在開始之前，而不是在出問題之後。

## 關鍵重點
- gstack 的角色分工設計解決了「單一巨型 CLAUDE.md 導致不同類型工作互相干擾」的問題
- `/office-hours` 的六個逼問問題強制把隱藏假設前置，避免在錯誤方向上快速前進
- `/investigate` 要求修 bug 前先完成根因分析的四個步驟，防止「修症狀而非原因」的常見模式
- 完整循環（Think → Plan → Build → Review → Test → Ship → Reflect）的每一步都是下一步的基礎，跳步的成本會在後面出現
- gstack 適合有一定規模的專案；小型專案應選擇性採用其機制，而不是整套搬入
