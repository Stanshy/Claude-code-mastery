## 正文

### 1. 背景：從高效率到精神疲憊的反思

David Chu 是前 ENS 團隊的軟體工程師。按照表面數字看，他的工作流非常成功：每天同時跑 3 至 5 個 Claude Code 實例，100% 程式碼由 Claude Code 生成，每天合併 2 至 4 個 PR。

但他發現自己每天下班時精神疲憊。

原因不是工作量太大，而是工作的性質：他花了大量時間在不斷切換終端機、審查輸出、重新下提示。這些動作本身不需要判斷力，但卻消耗了大量注意力。他感覺自己像是在「管理」Claude，而不是在「與 Claude 協作」。

這個挫折讓他開始思考一個更根本的問題：**Claude Code 的協作不應該只是「給任務、等輸出、給反饋」的循環，而應該是一個隨著時間演進、需要人類介入越來越少的系統。** 這個思考催生了五層進化模型。

---

### 2. 五層進化模型

| 層級 | 名稱 | 特徵 | 你在做什麼 |
|------|------|------|-----------|
| 第一層 | 對話式 | 輸入提示，Claude 回應 | 手動驅動每一步 |
| 第二層 | 配置式 | CLAUDE.md 引導，記憶保留 | 規則自動生效 |
| 第三層 | 自動化 | Hooks 強制，Skills 編碼 | 規則被保證執行 |
| 第四層 | 編排式 | 自訂 Agents 各司其職 | 專門角色處理專門問題 |
| 第五層 | 自主式 | Agent 團隊獨立運作 | 你只做最終 review |

每一層的進化都解決了前一層的具體問題：

**第一層到第二層：** 解決「每次任務都要重新解釋背景」的問題。CLAUDE.md 讓規則在 session 之間持久化，你不需要每次都重新說明「我們的程式碼風格是這樣的」。

**第二層到第三層：** 解決「Claude 理解規則但不持續遵守」的問題。Hooks 把規則從「Claude 應該記住的事」提升到「技術上必然發生的事」。Skills 把工作流程從「每次重新描述」變成「一個命令觸發完整流程」。

**第三層到第四層：** 解決「一個 Agent 處理所有類型任務導致專注度下降」的問題。當你的任務開始分化——有的需要安全視角、有的需要效能視角、有的需要用戶體驗視角——專門的 Agent 比通用的設定更可靠。

**第四層到第五層：** 解決「人類還是需要協調多個 Agent 之間的工作」的問題。當 Agent 之間的協作被系統化後，人類的角色從「協調者」退化為「最終審查者」。

進化不是強制的，也不是線性的。你可以在第二層運作很長時間，只要 CLAUDE.md 能有效解決你的問題。進化的觸發條件永遠是「當前層級無法可靠解決某類問題」。

---

### 3. 三條畢業路徑

「畢業」是 David Chu 用的比喻：當一條 CLAUDE.md 規則成熟到足以被更強的機制接管，它就從 CLAUDE.md 畢業，升級到 Hook、Skill 或 Agent。

判斷升級到哪裡，取決於問題的性質。

---

#### 路徑一：CLAUDE.md → Hook

**適用情境：** 當 Claude 不再持續遵守某條規則。

文字規則有一個固有局限：在長對話或複雜任務中，Claude 的注意力可能從規則上漂移。如果某條規則被違反一次、兩次、三次，這個規則就是 Hook 的候選。

**Hooks 的三個關鍵類型：**

- **PreToolUse**：在工具執行前介入。用於條件判斷——如果某個條件不滿足，阻止工具執行。例如：在任何 `Write` 工具執行之前，檢查目標路徑是否在允許的範圍內。
- **PostToolUse**：在工具執行後介入。用於回調驗證——如果工具輸出不符合預期，觸發修正流程。例如：在 `Bash` 工具執行後，檢查程式碼格式是否符合規範，若不符合自動執行格式化。
- **Stop**：在 Claude 準備結束任務時介入。用於品質關卡——確認所有必要步驟都已完成才允許結束。例如：在 Claude 準備結束之前，確認測試已通過。

**驗證器的數學：**

這是理解 Hook 價值的最重要概念。假設你有一個 5 步工作流，每一步 Claude 的成功率是 80%：

- 無驗證器：0.8 × 0.8 × 0.8 × 0.8 × 0.8 = 33% 整體成功率

加入一個驗證器（假設驗證器準確率 70%）後，Claude 有機會重試失敗的步驟，假設每步可重試 4 次：

- 有驗證器：1 - (1 - 0.7)^4 × 每步 ≈ 99% 整體成功率

這個數學說明了一件事：**一個好的驗證器，加上差的工作流，勝過好的工作流加上沒有驗證器。** 驗證器是 Claude Code 系統中槓桿率最高的單一投資。

---

#### 路徑二：CLAUDE.md → Skill

**適用情境：** 兩種情況需要把規則提升為 Skill：

1. **不常使用的工作流程**：如果某個工作流一個月只用一次，它的規則不應該常駐在 CLAUDE.md 中佔用 context 空間。把它寫成 Skill，只在需要時載入。

2. **模型未經訓練的知識**：如果你使用的是特定版本的框架、內部函式庫或非主流技術棧，Claude 的訓練資料中可能沒有足夠的相關知識。把這類知識封裝成 Skill，需要時精確注入。

**兩個具體範例：**

- **Dexie.js 設定 Skill**：如果你用 Dexie.js 做前端資料庫，Claude 對它的最佳實踐可能掌握不足。把「如何正確設定 Dexie.js schema、migration、和 reactive queries」寫成一個 Skill。每次涉及 Dexie.js 的任務，在任務描述前先執行這個 Skill。

- **團隊 UI 偏好 Skill**：如果你的團隊對 UI 元件有特定規範（例如：所有 modal 必須用 `<Dialog>` 元件、所有表單必須用 `react-hook-form`、所有錯誤狀態必須顯示 toast），把這些規範寫成一個 Skill，在前端任務開始前載入。

---

#### 路徑三：CLAUDE.md → Agent

**適用情境：** 當某類問題需要跨多個任務的持續專業判斷。

**Skill 和 Agent 的根本區別：**

- 為 Skill 定義**任務**：「執行這個工作流程」
- 為 Agent 定義**人設**：「你是一個擁有這些知識和判斷力的角色」

Skill 是靜態的流程，Agent 是動態的角色。Agent 可以在不同情境下應用它的角色知識，做出 Skill 做不到的情境判斷。

**兩個具體範例：**

- **security-reviewer Agent**：不只是一個安全檢查清單，而是一個持續從安全視角評估所有決策的角色。它知道你的系統架構，了解哪些部分是高風險的，在任何涉及認證、授權、資料處理的任務中都能提供專業判斷。

- **debugger Agent**：不只是「找 bug 的 Skill」，而是一個熟悉你程式碼庫、了解常見問題模式、能夠根據症狀快速縮小根因範圍的角色。它記得上週修過的類似問題，能夠把跨時間的 pattern 連結起來。

### 4. 三個畢業信號

David Chu 在實踐中識別出三個信號，當任何一個出現時，就是時候審視你的 CLAUDE.md 並考慮畢業：

**信號一：CLAUDE.md 達到約 50 行**

當 CLAUDE.md 超過 50 行，你需要開始問：哪些規則是每天必需的？哪些是只有特定情境才用到的？後者是 Skill 的候選。

50 行不是絕對的門檻，而是一個檢視點。這個長度的 CLAUDE.md 可能已經包含了足夠多的「場合特定規則」，值得整理一次。

**信號二：Claude 不再持續遵守某條規則**

這是最明確的信號。如果你發現自己在多個 session 中反覆提醒 Claude 同一件事，這條規則需要從文字層面升級到技術層面——變成 Hook。

**信號三：到了定期審查的時間點**

David Chu 建議設定固定的 CLAUDE.md 審查節奏，例如每兩週一次。這個定期審查的目的是主動識別哪些規則已經「成熟」到可以畢業，而不是等到問題發生才被動反應。

### 5. Boris 學派 vs RPI 學派的整合

對於 CLAUDE.md 應該有多「豐富」，社群中有兩種對立的觀點：

| 學派 | 觀點 | 適用階段 |
|------|------|---------|
| Boris Cherny | 不留情面地投資 CLAUDE.md，犯錯就加 | 早期，培養習慣 |
| RPI 學派 | CLAUDE.md 控制在 60 行內，超過 40% context 進入「笨蛋區」 | 成熟期，精簡優化 |
| 整合觀點（David Chu） | 先用 Boris 方法累積，規則成熟後畢業出去 | 全過程 |

**Boris 學派的核心主張：** 每次 Claude 犯錯就在 CLAUDE.md 加一條規則。不要吝嗇，不要猶豫。這是建立機構記憶的最快方式。

**RPI 學派的核心主張：** 過長的 CLAUDE.md 有反效果。當 CLAUDE.md 佔用超過 context window 的 40% 時，Claude 的整體表現下降——它進入了所謂的「笨蛋區」，所有能力都受到影響。CLAUDE.md 需要精簡，只保留最必要的規則。

這兩個觀點看起來矛盾，但 David Chu 的整合觀點解釋了它們其實在說不同的事：

Boris 說的是**早期**：先快速累積，建立從踩坑到規則的習慣。

RPI 說的是**成熟期**：當規則夠多時，需要定期精簡，把成熟的規則畢業出去，而不是讓它們永久佔用 context。

**Boris 的密集 CLAUDE.md 能運作的秘密武器：**

Boris 的 CLAUDE.md 之所以在 2,500 tokens 的情況下仍然有效，是因為他有一個強制執行 CLAUDE.md 本身的驗證器。這個驗證器確保 Claude 在每次任務開始時真正讀取並應用 CLAUDE.md 的規則，而不是形式上載入後就忽略。這個驗證器現在已成為 `/simplify` 指令的一部分。

**整合觀點的實踐：**

1. 用 Boris 方法累積規則（犯錯就加）
2. 定期審查，把穩定且高頻的規則升級為 Hook
3. 把場合特定的規則提取為 Skill
4. 把需要持續專業判斷的角色升級為 Agent
5. 保持 CLAUDE.md 作為「活規則」的精簡集合

---

### 6. 系統成長的方向

David Chu 的框架最終給出了一個很清晰的方向：

- **當 Claude 犯錯時，先將規則加進 CLAUDE.md**——讓錯誤只發生一次。
- **當 Claude 不再遵守規則時，將它升級為 Hook、Skill 或 Agent**——讓規則從文字升級到機制。
- **這樣能確保你的工具從實際遇到的摩擦中成長，而非過早優化**——系統的每一次進化都有真實需求驅動，不是理論設計。

這是一個反脆弱的系統設計方式：每次失敗都讓系統變得更強，而不是需要人工修復。

## 教學舉例

### 場景：一條 CLAUDE.md 規則從建立到畢業成 Hook 的完整生命週期

**起點：第一次踩坑（第 0 天）**

你在用 Claude Code 開發一個 Node.js API。某一天，Claude 在處理一個複雜的資料庫查詢任務時，直接把 SQL 查詢字串拼接了起來：

```javascript
// Claude 生成的程式碼
const query = `SELECT * FROM users WHERE id = ${userId}`;
db.query(query);
```

這是一個 SQL injection 漏洞。你抓到了，在 code review 時修正。

**第一步：加入 CLAUDE.md（第 0 天，花費 2 分鐘）**

修正後，你立刻在 CLAUDE.md 加入一條規則：

```
## 安全規則
- 所有資料庫查詢必須使用參數化查詢（parameterized queries）
  - PostgreSQL: 使用 $1, $2 佔位符
  - MySQL: 使用 ? 佔位符
  - 絕對禁止字串拼接 SQL
```

接下來兩週，這條規則有效運作。Claude 在生成資料庫查詢時都使用參數化查詢。

**信號出現：規則開始被違反（第 15 天）**

你的 CLAUDE.md 已經成長到 45 行，包含了多個規則。在一個特別複雜的任務中——涉及多個資料表 join 和動態 filter 條件——Claude 又一次生成了字串拼接的查詢。

你修正後再次提醒，但你意識到這條規則正在進入「不再持續遵守」的狀態。這是第二個畢業信號。

你決定讓這條規則畢業。

**第二步：設計 Hook（第 15 天，花費 30 分鐘）**

你建立一個 PostToolUse Hook，在每次 `Write` 工具執行（也就是 Claude 寫入檔案）後觸發：

```typescript
// .claude/hooks/sql-injection-guard.ts
export async function onPostToolUse(context: ToolUseContext) {
  if (context.tool.name !== 'Write') return;

  const content = context.tool.output.content as string;

  // 偵測字串拼接 SQL 的模式
  const sqlInjectionPatterns = [
    /`[^`]*SELECT[^`]*\$\{/i,           // 模板字串中的 SELECT
    /['"][^'"]*SELECT[^'"]*' \+/i,       // 字串連接的 SELECT
    /query\s*=\s*['"`][^'"`]*\+\s*\w/i, // query = "..." + variable
  ];

  const violations = sqlInjectionPatterns
    .filter(pattern => pattern.test(content))
    .length;

  if (violations > 0) {
    return {
      action: 'block',
      message: `偵測到潛在的 SQL injection 模式。

必須使用參數化查詢：
- PostgreSQL: db.query('SELECT * FROM users WHERE id = $1', [userId])
- MySQL: db.query('SELECT * FROM users WHERE id = ?', [userId])

請修正後重新提交。`
    };
  }
}
```

**第三步：測試 Hook（第 15 天，花費 15 分鐘）**

你寫一個測試任務，故意讓 Claude 生成字串拼接的查詢，確認 Hook 確實攔截並提示正確做法。你也測試一個正確的參數化查詢，確認 Hook 不會誤攔截。

兩個測試都通過。Hook 正式上線。

**第四步：更新 CLAUDE.md（第 15 天，花費 5 分鐘）**

你把 CLAUDE.md 中的那條安全規則改為：

```
## 安全規則
- SQL injection 防護已由 PostToolUse Hook 自動執行（sql-injection-guard.ts）
  - 字串拼接 SQL 會被自動攔截，無需手動提醒
```

這條記錄的目的是讓未來的你（或任何閱讀 CLAUDE.md 的人）知道：這個規則已經畢業了，不需要在 CLAUDE.md 中維持詳細說明，因為執行已由 Hook 保證。

CLAUDE.md 從 45 行減少到 42 行。

**結果：系統比之前更強**

- 這條規則從「Claude 應該記住的事」升級到「技術上必然發生的事」
- 即使 Claude 的注意力在複雜任務中漂移，Hook 仍然保證規則被執行
- CLAUDE.md 沒有因為這條規則而膨脹，反而因為畢業而精簡了

**15 天後（第 30 天）的 review**

你做定期 CLAUDE.md 審查。Hook 執行了 23 次，其中攔截了 4 次違規。這 4 次都是在複雜的多表 join 任務中出現的，正是在長 context 下最容易遺忘規則的場景。

你確認 Hook 的設計是正確的。你也注意到 Hook 有一次誤攔截了一個使用模板字串但不是 SQL 的地方——你更新 pattern 規則，讓它更精確。

這就是一條規則的完整進化生命週期：從踩坑 → CLAUDE.md 規則 → Hook，每一步都由真實的需求驅動，每一步都讓系統變得更可靠。

## 關鍵重點
- 五層進化模型的每次升級都由具體問題驅動，不是理論設計的結果；進化的觸發條件是「當前層級無法可靠解決某類問題」
- PostToolUse Hook 加上驗證邏輯的組合，能把整體成功率從 33% 提升到接近 99%，是系統中槓桿率最高的單一投資
- Skill 和 Agent 的根本區別：為 Skill 定義任務，為 Agent 定義人設
- Boris 學派（快速累積）和 RPI 學派（精簡 context）並不矛盾：前者適用於早期建立習慣，後者適用於成熟期精簡，畢業機制連接了兩者
- 系統應該從實際遇到的摩擦中成長：先加 CLAUDE.md，不再遵守時升級為 Hook/Skill/Agent，避免過早優化
