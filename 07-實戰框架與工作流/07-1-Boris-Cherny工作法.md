# 07-1 Boris Cherny 工作法
> Claude Code 創建者如何在 30 天內完成 259 個 PR。

## 學習目標
- 理解 Boris Cherny 的「平行 session」策略，以及它如何消除人類等待時間
- 掌握 CLAUDE.md 作為「機構記憶」的具體運作方式
- 學會「探索 → 計畫 → 實作 → 提交」四步循環的每個節點該做什麼
- 理解為什麼選用較強模型在整體效率上是合理的投資

---

## 正文

### 1. 背景：一個出乎意料平凡的故事

Boris Cherny 是 Claude Code 的創建者。他在 30 天內完成了 259 個 PR，每一行程式碼由 Claude 撰寫，產出約 40,000 行。當人們問他靠什麼「秘訣」時，他的回答出乎所有人的意料：「這套方式出乎意料地平凡。」

這句話值得深思。259 個 PR 並不是靠某個神奇咒語達成的，而是靠三件事的持續疊加：降低每次變更的風險、消除人類等待時間、持續累積品質知識。這三件事都可以被系統化，而 Boris 的 13 項核心技巧正是這個系統的具體表達。

本節聚焦最重要的 7 個技巧，從「為什麼這樣做」出發，再談「怎麼做」。

---

### 2. 技巧一：堅持用 Opus（或當下最強的模型）

**為什麼：** 修正成本遠超運算成本。當 Claude 用較弱的模型生成了錯誤的實作，你需要花時間：閱讀程式碼、識別問題、重新下提示、再次等待輸出、再次驗證。這個迴圈花費的時間，是 Opus 和 Sonnet 差價的 10 至 20 倍。用便宜模型省下的 API 費用，會在 debug 迴圈中以十倍的時間成本還回來。

**怎麼做：** 對於任何會進入 code review 的工作，預設選用最強的可用模型。只有在確定是探索性、低風險的任務時，才考慮降級。不要在結果上省錢。

---

### 3. 技巧二：平行 5+5 模式

**為什麼：** 人類在等待 Claude 回應時是閒置的。如果你同時只跑一個 session，你的有效工作率受限於 Claude 的生成速度，而不是你的判斷速度。平行化把人類的瓶頸從「等待」移到「判斷」，這才是對人類時間最正確的使用方式。

**怎麼做：** 同時跑 5 個本地 session（本機終端機）加上 5 至 10 個線上 remote session（例如 GitHub Actions 或雲端環境）。使用 macOS 系統通知協調切換——當 Claude 完成一個任務時，桌面會彈出通知，你切過去 review，確認沒問題後啟動下一個任務，再切回其他 session。這讓你的注意力永遠在「最需要人類判斷的地方」，而不是在等待游標閃爍。

---

### 4. 技巧三：每 PR 保持聚焦

**為什麼：** 大型 PR 是品質的敵人。一個包含 800 行變更、橫跨 5 個功能的 PR，review 者無法深入思考每個部分，合併後若出問題難以回滾，偵錯時也難以定位變更點。小 PR 讓每個決策都清晰可見。

**怎麼做：** 強制遵守兩個規則：一個 PR 只包含一個邏輯變更，且行數在 200 行以內。如果一個功能需要更多行，把它拆成多個 PR，按依賴順序合併。這個紀律在短期看起來是「多做事」，但它讓整個開發週期的速度提升，因為每個節點都更安全。

---

### 5. 技巧四：CLAUDE.md 機構記憶

**為什麼：** 每次 Claude 犯同一個錯誤，你就浪費了一次 debug 迴圈。如果這個錯誤是「可以預防的」，而你沒有把預防規則寫下來，你就在接受一個本可避免的重複成本。CLAUDE.md 是讓錯誤只發生一次的機制。

**怎麼做：** 每次 Claude 犯錯，在修正後立刻把預防規則加進 CLAUDE.md。Boris 的 CLAUDE.md 約 2,500 tokens，內容是全團隊在 code review 過程中持續迭代的結果。這不是一個人寫好的文件，而是從真實踩坑中長出來的知識庫。

---

### 6. 技巧五：探索 → 計畫 → 實作 → 提交

**為什麼：** 直接跳到實作是最常見的效率陷阱。沒有探索，Claude 可能對程式碼庫的現有結構做出錯誤假設。沒有計畫，實作過程中會出現方向性問題，需要回頭重來。這個四步循環強制把思考成本前置，讓實作過程更直線。

**怎麼做：**
- **探索**：讓 Claude 讀取相關檔案、理解現有架構，不做任何修改。
- **計畫**：切換到 Plan Mode（`claude --plan`），讓 Claude 列出實作步驟，你審查並確認方向。
- **實作**：切回 Normal Mode，按計畫執行。
- **提交**：用 `/commit` 或標準 git flow 建立 PR，保持 PR 聚焦。

---

### 7. 技巧六：給 Claude 驗證方式

**為什麼：** Claude 生成程式碼後，如果沒有任何回饋機制，它無法判斷輸出是否正確。驗證方式是 Claude 自我修正的觸發器。沒有驗證，Claude 會假設它的輸出是正確的，即使它是錯的。

**怎麼做：** 提供至少一種驗證方式：測試腳本（跑後看輸出）、截圖（視覺驗證）、預期輸出範例（對比驗證）。告訴 Claude：「完成後跑這個測試，如果失敗，自己修正直到通過。」這是整個系統中槓桿率最高的做法，因為它把修正迴圈的人力成本降到最低。

---

### 8. 技巧七：設定出乎意料地簡單

**為什麼：** 很多人假設 Boris 有一套複雜的自訂系統才能達到這樣的產出。這個假設是錯的，而且危險——它讓人覺得「我還沒準備好」，推遲了開始行動的時間。

**怎麼做：** Boris 說他的配置是「vanilla」——沒有複雜的自訂 Hook、沒有精密的 Agent 架構。核心就是三件事：CLAUDE.md（機構記憶）+ 好的 prompt（清晰的任務描述）+ 驗證方式（讓 Claude 自我檢查）。你今天就可以用這三件事開始。

---

### 9. 為什麼這套方法有效

Boris 的方法之所以有效，不是因為某個神奇技巧，而是因為它同時解決了三個獨立的效率瓶頸：

**降低每次變更的風險（小 PR）：** 每個 PR 的範圍都小到可以被快速、完整地審查。錯誤被限制在小範圍內，回滾成本低。

**消除等待時間（平行 session）：** 人類的注意力永遠在有意義的工作上，而不是在等待 Claude 的游標。生產效率不再受限於單一 session 的回應速度。

**持續累積品質（CLAUDE.md 機構記憶）：** 每次 debug 的結果都轉化為預防規則，系統隨時間變得越來越少犯錯。這是一個持續改善的飛輪，不是靜態的設定。

---

### 4. Boris 方法 vs OpenAI 方法的對比

Boris 的方法不是唯一的高產出工作法。OpenAI 的 Codex 團隊在 5 個月內用零手寫程式碼產出了 100 萬行、1,500 個 PR。兩套方法的對比揭示了不同規模下的最佳實踐：

| 面向 | Boris Cherny | OpenAI Codex 團隊 |
|------|-------------|------------------|
| 人類角色 | 寫 prompt + review 每個 PR | 設計環境 + 指定意圖 |
| 程式碼產出 | 每行由 Claude 寫 | 每行由 Codex 寫 |
| Review | 人類 review 每個 PR | Agent 互相 review，人類選擇性介入 |
| 合併策略 | 小 PR（200 行內）+ 嚴格 review | 快速合併，修正比等待便宜 |
| 清理 | CLAUDE.md 持續迭代 | GC Agent 自動開重構 PR |
| 吞吐量 | 每人每天 ~8 PR（259/30） | 每人每天 3.5 PR |
| 核心差異 | 個人工作法，低門檻 | 團隊系統設計，需基礎設施投入 |

兩者不矛盾。Boris 的方法更適合個人或小團隊快速上手——你今天就能開始用；OpenAI 的方法更適合有資源投入 Harness 基礎設施的團隊——需要數週到數月的建設期。

## 教學舉例

### 場景：用 Boris 工作法開發一個「用戶標籤管理」功能

假設你正在開發一個 SaaS 平台，需要新增「用戶標籤管理」功能，允許管理員為用戶打標籤、查詢特定標籤的用戶列表，以及移除標籤。

**準備階段（開始前）**

你的 CLAUDE.md 已經記錄了幾條規則，例如：「所有資料庫查詢必須使用參數化查詢，不得字串拼接」、「API endpoint 必須通過現有的 `authMiddleware` 驗證」。這些是之前踩坑後加進去的規則。

**步驟一：探索**

你開一個新的 Claude Code session，給出這個提示：

「請讀取 `src/models/User.ts`、`src/routes/users.ts`、`src/middleware/auth.ts`，了解現有的用戶資料結構和路由模式。不要做任何修改，只需描述你觀察到的結構。」

Claude 回報：用戶模型目前有 `id`、`email`、`role` 三個欄位，路由使用 Express，中間件已有 `authMiddleware` 包裝。沒有現有的標籤系統。

**步驟二：計畫**

切換到 Plan Mode：

「根據你觀察到的結構，列出實作『用戶標籤管理』功能需要的所有步驟。功能包含：為用戶打標籤、查詢特定標籤的用戶、移除標籤。」

Claude 列出計畫：
1. 新增 `Tag` 資料模型（`id`、`name`、`userId`）
2. 新增資料庫 migration
3. 新增三個 API endpoint（POST `/tags`、GET `/tags/:tag`、DELETE `/tags/:id`）
4. 為所有 endpoint 套用 `authMiddleware`
5. 新增對應的 unit test

你審查計畫，確認方向正確，補充一條：「所有查詢需要分頁，使用現有的 `paginateQuery` 工具函式。」

**步驟三：開平行 session**

你注意到這個功能可以拆成兩個獨立的 PR：
- PR 1：資料模型 + migration（無依賴，可先合併）
- PR 2：API endpoint + test（依賴 PR 1）

你開兩個 session。Session A 處理 PR 1，Session B 在旁待命（你先給它探索任務，等 PR 1 合併後再啟動實作）。

你的 macOS 通知設定好了，Session A 完成時會彈通知。

**步驟四：給 Claude 驗證方式**

在 Session A 的提示中加上：「完成後執行 `npm run migrate:test` 和 `npm test -- --testPathPattern=tag`，若測試失敗，自行修正直到全部通過。」

Session A 開始工作。你切去做其他事。

通知彈出。你回來看 Session A 的輸出：測試全過，PR 1 的變更是 `src/models/Tag.ts`（43 行）和 `migrations/001_add_tags.sql`（12 行），共 55 行。符合 200 行以內的規則。

**步驟五：review 與合併**

你快速 review 55 行的變更：資料模型結構正確，migration 有 rollback 腳本，沒有字串拼接查詢。5 分鐘完成 review，合併 PR 1。

**步驟六：CLAUDE.md 更新**

你在 review 時發現 Claude 在 Tag 模型中沒有加 `createdAt` 欄位，雖然這次任務規格沒提，但你知道所有資料表都應該有這個欄位。你在 CLAUDE.md 加上一條：「所有新增的資料模型必須包含 `createdAt` 和 `updatedAt` 欄位，使用 `timestamp with time zone` 類型。」

下次 Claude 建新模型時，這條規則會自動生效。

**結果**

這個功能從探索到兩個 PR 合併，耗時約 3 小時，其中你的主動工作時間（思考、review、決策）約 45 分鐘。其餘時間你在處理其他任務，因為平行 session 讓你的等待時間幾乎為零。

這就是 Boris 工作法的日常面貌：不是魔法，而是把對的事情做到位的系統。

## 關鍵重點
- Opus 的額外費用遠低於 debug 迴圈消耗的時間成本，用最強模型做正式工作是理性選擇
- 平行 session 把人類的瓶頸從「等待」移到「判斷」，是提升有效工作率最直接的方式
- 每 PR 只做一個邏輯變更且在 200 行以內，讓 review、合併、回滾都更安全
- CLAUDE.md 是從真實踩坑長出來的機構記憶，每次錯誤都應該轉化成一條預防規則
- 給 Claude 驗證方式（測試、截圖、預期輸出）是整個系統中槓桿率最高的單一做法

## 自我檢核
- 如果你今天要開始用 Boris 工作法，你現有的 CLAUDE.md 裡有多少條規則是從真實踩坑中提取的？哪些只是「感覺應該寫進去」的規則？
- 你上次的 PR 有多少行？如果超過 200 行，哪個部分可以拆出去成為獨立的 PR？
- 當 Claude 在某個 session 工作時，你在做什麼？如果答案是「等待」，平行化哪些任務可以消除這個等待？
