# 02-2 設定檔與權限系統
> 用 settings.json 預先定義操作邊界，讓 Claude Code 在受控範圍內自主工作。

## 學習目標
- 理解 Claude Code 設定的 6 層優先級結構
- 掌握 settings.json 中 allow / deny 規則的語法
- 能夠為專案配置合理的權限邊界
- 了解五種權限模式的差異與適用情境

---

## 為什麼設定檔和權限很重要

沒有設定的 Claude Code，每次操作都會彈出權限確認。你會花大量時間在按「允許」。更嚴重的是，Claude 可能執行你不希望它做的事——刪除檔案、修改 `.env`、force push。設定檔和權限系統的目的是：預先定義什麼可以做、什麼不能做，減少手動介入，同時在自動化流程中保護關鍵資源不被誤操作。

設定得當的工作環境，讓 Claude 在跑測試、格式化程式碼、提交 git 時完全不需要打斷你，同時讓它無法觸碰 `.env` 或執行破壞性 git 指令。這是「讓 AI 高效工作」和「保持人類掌控」的平衡點。

---

## 分層優先級

Claude Code 的設定有 6 層，優先級從高到低：

| 優先級 | 來源 | 說明 |
|--------|------|------|
| 最高 | 管理策略（managed） | 企業管理員統一設定，使用者無法覆蓋 |
| ↓ | CLI 參數（--flag） | 當次會話有效，結束即失效 |
| ↓ | .claude/settings.local.json | 本地專案設定（應加入 gitignore） |
| ↓ | .claude/settings.json | 共享專案設定（commit 到 git） |
| ↓ | ~/.claude/settings.json | 使用者級別全域設定 |
| 最低 | 插件配置 | 插件預設提供的設定 |

實務上最常用的是 `.claude/settings.json`（團隊共享的基礎規則）和 `.claude/settings.local.json`（個人覆蓋，不進版控）。前者定義「專案允許什麼」，後者定義「我個人的環境變數或偏好」。

---

## settings.json 關鍵欄位

以下是最實用的設定結構，不需要記住所有欄位，先理解這個核心框架：

```json
{
  "permissions": {
    "allow": [
      "Bash(npm run *)",
      "Bash(npx jest *)",
      "Bash(git commit *)",
      "Bash(git push *)"
    ],
    "deny": [
      "Bash(rm -rf *)",
      "Bash(git push --force *)",
      "Edit(.env*)"
    ]
  },
  "env": {
    "NODE_ENV": "development"
  }
}
```

`allow` 陣列列出「預先批准、不需確認」的操作。每次 Claude 嘗試執行這些指令時，會直接執行而不彈出提示。

`deny` 陣列列出「明確禁止、即使被要求也拒絕」的操作。這是安全底線，不受其他設定影響。

`env` 區塊注入環境變數，讓 Claude 在執行指令時能取得正確的執行環境。

---

## 五種權限模式

除了靜態的 allow / deny 規則，Claude Code 還有五種整體模式，控制未被明確列出的操作如何處理：

| 模式 | 行為 | 適用情境 |
|------|------|----------|
| default | 每個工具調用都提示確認 | 日常標準使用 |
| plan | 唯讀探索，不允許任何修改 | 分析 codebase、理解現有架構 |
| acceptEdits | 自動接受所有檔案編輯 | 明確任務（如重構特定模組） |
| dontAsk | 自動拒絕未在 allow 中的工具 | 限定範圍的自動化任務 |
| bypassPermissions | 跳過所有提示 | 僅在隔離沙盒環境中使用 |

切換方式：對話介面中按 `Shift+Tab` 循環切換，或輸入 `/permissions` 選擇。

`bypassPermissions` 模式很危險，不應在本機開發環境使用。即使在這個模式下，Claude Code 也會對以下目錄保留確認提示：`.git/`、`.claude/`、`.vscode/`、`.idea/`。這些是受保護目錄，不受模式設定影響。

---

## allow / deny 規則語法

規則語法是「工具名稱（參數模式）」的格式，支援以下四種匹配方式：

| 模式 | 範例 | 匹配說明 |
|------|------|----------|
| 精確匹配 | `Bash(npm run test)` | 只匹配完全相同的指令 |
| 前綴通配 | `Bash(npm *)` | npm 開頭的所有指令 |
| 路徑模式 | `Edit(.env*)` | .env、.env.local、.env.production 等 |
| MCP 工具 | `mcp__github__*` | GitHub MCP 提供的所有工具 |

設計規則時有一個重要原則：allow 可以寬，deny 要精確。

allow 寬一點（如 `Bash(npm run *)`）的壞處是偶爾允許非預期的 npm script，但通常可接受。deny 如果太寬（如 `Bash(git *)`），會把 `git status`、`git log` 這類安全操作也一起擋掉，造成不必要的摩擦。deny 應該只針對破壞性操作，例如 `Bash(git push --force *)` 和 `Bash(git reset --hard *)`，而不是整個 git 工具集。

## 實作範例：為 AI Dev Assistant 配置 settings.json

> 主案例延續：在 ai-dev-assistant 專案中建立完整的權限邊界設定，讓 Claude 能自主處理開發任務，同時保護關鍵資源。

**目標**：建立 `.claude/settings.json`，允許日常開發操作，阻擋危險指令；並建立個人本地設定 `.claude/settings.local.json`。

**前置條件**：已完成 01-2（專案骨架建立）和 02-1（CLAUDE.md 配置），專案根目錄存在 `package.json` 和 `.gitignore`。

**步驟**：

1. 在專案根目錄建立 `.claude` 目錄：
   ```bash
   mkdir -p .claude
   ```

2. 建立 `.claude/settings.json`，寫入以下內容：
   ```json
   {
     "permissions": {
       "allow": [
         "Bash(npm run *)",
         "Bash(npx jest *)",
         "Bash(npx tsc *)",
         "Bash(npx prettier *)",
         "Bash(npx eslint *)",
         "Bash(git add *)",
         "Bash(git commit *)",
         "Bash(git push *)",
         "Bash(git status)",
         "Bash(git diff *)",
         "Bash(git log *)"
       ],
       "deny": [
         "Bash(rm -rf *)",
         "Bash(git push --force *)",
         "Bash(git reset --hard *)",
         "Edit(.env*)",
         "Edit(.claude/settings.json)"
       ]
     }
   }
   ```

3. 驗證 JSON 語法正確（在寫入後立即執行）：
   ```bash
   node -e "JSON.parse(require('fs').readFileSync('.claude/settings.json','utf8')); console.log('JSON valid')"
   ```

4. 建立 `.claude/settings.local.json`（個人設定，不進 git）：
   ```json
   {
     "env": {
       "NODE_ENV": "development"
     }
   }
   ```

5. 將本地設定加入 `.gitignore`，開啟 `.gitignore` 並加入：
   ```
   .claude/settings.local.json
   ```

6. 啟動 Claude Code，嘗試執行 `npm test`——Claude 應直接執行，不彈出確認視窗。

7. 在對話中要求 Claude 修改 `.env`——Claude 應回報操作被拒絕，不執行修改。

**預期結果**：
- `npm run *` 和 `git add / commit / push / status / diff / log` 等操作不再需要手動確認
- 嘗試執行 `rm -rf`、`git push --force`、`git reset --hard` 時被系統拒絕
- 嘗試編輯 `.env` 或 `.env.local` 時被系統拒絕
- `.claude/settings.local.json` 出現在 `.gitignore` 中，不會被 `git status` 追蹤

**常見錯誤**：

- **JSON 語法錯誤（缺逗號或多餘逗號）**：Claude Code 啟動時會報 `parse error`，無法載入設定。用上述 node 指令驗證，或安裝 `jq` 後執行 `jq . .claude/settings.json`。

- **deny 規則太寬**：例如寫 `Bash(git *)` 會把 `git status`、`git log` 全部擋掉，造成 Claude 無法查看 git 狀態。deny 要針對具體危險操作，不要用過於寬泛的通配符。

- **忘記把 settings.local.json 加入 .gitignore**：個人環境變數或 API 金鑰可能洩漏進版控。每次建立 local 設定後，立即確認 `.gitignore` 已更新。

- **allow 規則寫了但仍彈出確認**：確認規則語法正確，工具名稱區分大小寫（`Bash` 不是 `bash`），且參數模式需與實際指令前綴完全匹配。

---

## 關鍵重點

- settings.json 有 6 層優先級，`.claude/settings.json`（共享）和 `.claude/settings.local.json`（個人）是日常最常用的兩層
- allow 規則讓預批准的操作免確認，deny 規則設定不可逾越的安全底線
- deny 要精確（針對危險操作），allow 可以寬（允許整類操作）
- 即使在 bypassPermissions 模式下，`.git/` 和 `.claude/` 仍是受保護目錄
- settings.local.json 必須加入 .gitignore，避免個人環境設定或敏感資訊進入版控
