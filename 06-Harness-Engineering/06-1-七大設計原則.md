# 06-1 七大設計原則
> 工具本身不構成系統——七大設計原則讓你決定何時用哪個工具、用多少、何時移除。

## 學習目標
- 理解為什麼模型周圍的系統，而非模型本身，才是 Agent 可靠性的瓶頸
- 掌握七大設計原則的定義，以及每條原則在 Claude Code 中的具體對應機制
- 能辨識每條原則「做對」與「做錯」的具體樣子
- 能用七大原則回顧並審視自己建立的 Harness 是否完整

## 正文

### 1. 為什麼需要設計原則

到目前為止，你已經學了 CLAUDE.md、Hooks、Skills、Subagents、MCP、Worktree。這些是工具。但工具本身不構成系統——你需要一套設計原則來決定什麼時候用哪個工具、用多少、什麼時候移除。這就是 Harness Engineering。

「Harness」這個詞來自賽車工程：安全帶不是引擎，但沒有安全帶，引擎再強也無法讓車手發揮極限。Claude Code 的 Harness 同理——CLAUDE.md、Hooks、Skills 這些組件不是讓 Claude 更聰明，而是讓它的能力可以被可靠地、可重複地引導到正確方向。

**核心主張**：Agent 的可靠性瓶頸不在模型，在模型周圍的系統。

LangChain 在 2023-2024 年間反覆被社群詬病的問題，不是 GPT-4 不夠強，而是框架在上下文管理、工具呼叫邏輯、錯誤處理上的設計缺陷。相對地，Anthropic 的工程師在內部實驗中多次觀察到：同一個模型，只優化周圍的 prompt 結構、工具設計、驗證循環，就能讓任務成功率從 40% 提升到 80% 以上。模型沒有換，Harness 換了。

這個觀察產生了七大設計原則。每一條原則都對應一類真實失敗模式，並在 Claude Code 的機制中有明確的落地方式。

---

### 2. 七大原則逐一展開

#### 原則 1：上下文架構（Context Architecture）

**一句話定義**：給地圖，不給百科全書。Agent 按需查找，而非預載一切。

**問題根源**：語言模型的 context window 是有限資源。把所有可能相關的資訊塞進 context，不是讓 Claude 更強，而是讓真正重要的資訊被稀釋，同時增加推理成本與錯誤率。Claude 在一個 50 行的精準 context 裡表現，遠勝於在一個 2000 行的混雜 context 裡。

**在 Claude Code 中的對應**：
- `CLAUDE.md`：作為索引（< 200 行），告訴 Claude「哪裡有什麼」，而非把所有內容直接放入
- `Skills`：按需載入（`/skill-name`），只在需要時引入特定知識區塊
- `@` 匯入：在對話中即時引用特定文件，不預先載入

**做對的樣子**：CLAUDE.md 共 50 行，內容是專案架構說明、重要規則的條列，以及深層文件的引用路徑。當需要 API 規範時，Claude 執行 `@docs/api-spec.md` 來讀取。

**做錯的樣子**：把所有 API 文件、架構圖、三個月前的 ADR（Architecture Decision Record）、完整的規範全部貼進 CLAUDE.md，導致 CLAUDE.md 達到 3000 行，Claude 每次都要在雜訊中找真正需要的資訊。

---

#### 原則 2：架構約束（Architectural Constraints）

**一句話定義**：用工具強制執行規則，不靠 prompt 建議。

**問題根源**：Prompt 是建議，不是約束。「請不要做 X」這類指令在壓力情境下（例如 Claude 嘗試多次後依然失敗，開始嘗試捷徑）往往會被忽略。工具層的強制執行才是真正的約束——Hook 回傳 `exit 2`，Claude 物理上無法繼續。

**在 Claude Code 中的對應**：
- `Hooks`：PreToolUse 攔截危險操作（在工具執行前阻擋），PostToolUse 回調驗證結果
- `settings.json` 的 deny 規則：直接禁止特定 CLI 工具或參數

**做對的樣子**：Hook 攔截 `git push --force`，腳本檢測到 `--force` 參數後直接 `exit 2`，並輸出「Force push 已被 Harness 禁止，請使用 --force-with-lease 並說明原因」。Claude 收到阻擋訊息，無法繞過。

**做錯的樣子**：CLAUDE.md 裡寫「請不要 force push，這很危險」。Claude 在正常情況下會遵守，但當它嘗試了五次 rebase 都失敗、認為這是最後手段時，它還是可能選擇 force push。Prompt 建議在壓力下失效。

---

#### 原則 3：自驗證循環（Self-Validation Loop）

**一句話定義**：攔截死循環，並強制退出前驗證。

**問題根源**：Claude 說「完成了」，不等於真的完成了。模型可能在最後一步犯錯、忘記跑測試、或在修改中引入新 bug。如果沒有強制的退出驗證，「Agent 完成」等於「Agent 聲稱完成」。

**在 Claude Code 中的對應**：
- `Stop Hook Validator`：在 Claude 準備結束對話前，自動執行預定義的驗證腳本（例如 `npm test && npm run lint`）。若驗證失敗，Claude 被迫重新進入工作循環，而非直接交付有問題的結果。

**做對的樣子**：Stop Hook 執行 `npm test && npm run lint`。測試有一個 case 失敗，Hook 回傳失敗狀態，Claude 看到具體的錯誤輸出後修復問題，再次嘗試退出，這次驗證通過。

**做錯的樣子**：Claude 修改了三個檔案後說「任務完成，我已更新相關邏輯」。你信任它，直接 merge。CI 跑測試，發現第二個檔案引入了型別錯誤。修復代價是你的時間，不是 Claude 的。

---

#### 原則 4：上下文隔離（Context Isolation）

**一句話定義**：子 Agent 當防火牆，互不污染。

**問題根源**：長對話會累積中間產物——Claude 的推理過程、嘗試失敗的結果、臨時假設。這些中間產物會佔用 context、干擾後續判斷、甚至讓 Claude 在「既有思路」下犯更多錯誤。複雜任務的解法不是給 Claude 更長的 context，而是讓它在乾淨的環境中工作。

**在 Claude Code 中的對應**：
- `Subagents`：每個 Subagent 有獨立的 context，調查或生成工作完成後，只有精簡的結果回傳給主對話
- `Worktree`：Subagent 在獨立的 Git worktree 中操作，檔案系統層面的隔離，不影響主分支

**做對的樣子**：需要調查一個複雜的 codebase 安全問題，委派給 Security Reviewer Subagent。Subagent 在獨立 context 中閱讀五十個檔案、建立分析報告。主對話只收到一份三百字的結論，context 保持乾淨。

**做錯的樣子**：所有工作都在主對話中完成。Claude 在同一個 context 裡調查安全問題、修改程式碼、生成報告、再次修改。到了第 30 輪對話，context 裡充滿早已過時的中間推理，Claude 開始回應中前後矛盾。

---

#### 原則 5：熵管理（Entropy Management）

**一句話定義**：系統跑越久越亂，需要主動維護。

**問題根源**：每次踩坑後加一條規則，這是對的。但如果只加不刪，三個月後你有 30 條 Hook，有些互相衝突，有些對應的問題早已被新模型內建解決，有些根本從未被觸發。系統熵增是必然的——Harness 需要定期的主動維護。

**在 Claude Code 中的對應**：
- `Auto Memory`：Claude Code 自動記錄對話中的重要學習，避免同樣問題重複發生
- 踩坑 → 規則閉環：每個真實踩坑對應一條具體規則（Hook / Skill / CLAUDE.md 條目）
- 定期 CLAUDE.md 審計：每月或每季回顧，移除過時條目，整合重複規則

**做對的樣子**：每週五花 15 分鐘 review 踩坑日誌。本週 Claude 兩次在沒有測試的情況下提交 PR，提煉為 PreToolUse Hook，攔截沒有對應測試檔案的 `git commit`。同時發現三週前加的一條規則——「不要使用 `var`」——已被 ESLint 規則覆蓋，刪除。

**做錯的樣子**：踩坑時加規則，從不刪除。三個月後 CLAUDE.md 有 150 行，其中 60 行是過時或重複的。Hooks 目錄有 12 個腳本，有兩個從未被觸發，有一對腳本邏輯互相矛盾。系統越來越重，問題越來越難追蹤。

**進階做法（OpenAI 模式）**：不靠人工定期審計，而是部署背景運行的自動化 Agent：

- **Garbage Collection Agent**：週期性掃描 codebase 偏差（命名不一致、重複邏輯、違反架構規範），自動開重構 PR。多數能在一分鐘內 review 並自動合併。OpenAI 團隊初期每週五花 20% 時間手動清理「AI slop」，後來用自動化 Agent 取代。
- **Doc-gardening Agent**：掃描過時文件，檢查文件與程式碼的一致性，自動提交修復 PR。

這是熵管理的最終形態：人類品味只需捕獲一次（寫成規則），然後在每一行程式碼上持續自動執行。

---

#### 原則 6：可拆卸性（Removability）

**一句話定義**：Harness 必須模組化，更強的模型可移除部分組件。

**問題根源**：模型能力在快速演進。Claude 3 需要的很多 Harness，Claude 4 可能已經內建處理了。如果你的 Harness 是一個不可分割的整體，你就無法在模型升級後移除已不必要的部分，最終系統越來越重，維護成本越來越高。

**在 Claude Code 中的對應**：
- 每個 Hook 是獨立的 shell 腳本，刪除單一檔案不影響其他 Hook
- 每個 Skill 是獨立的 Markdown 檔案，移除後 Claude 退回到預設行為
- 每個 Agent 是獨立的 prompt 檔案，停用不影響主 Harness

**做對的樣子**：每月審計——「這個 Hook 最近 30 天有被觸發嗎？」「這個 Skill 對應的能力，新版 Claude 能自己處理了嗎？」發現 `auto-add-semicolons.sh` 在 Prettier 導入後從未觸發，直接刪除。系統輕了一點，沒有任何其他東西受影響。

**做錯的樣子**：把所有約束邏輯、所有 Agent 設定、所有特殊規則都寫進一個巨大的 `settings.json`，邏輯互相依賴。想移除其中一條規則，需要閱讀整個檔案確認影響範圍，風險太高，最後選擇什麼都不動。系統只能增長，永遠無法減輕。

---

#### 原則 7：Agent 可讀性（Agent Legibility）

**一句話定義**：Codebase 首先為 Agent 的可讀性最佳化，而非人類。

**問題根源**：從 Agent 的角度看，任何它在執行時無法在 context 中存取的東西，等同於不存在。存在於 Google Docs、Slack 對話、人腦中的知識，對 Agent 是不可存取的。就像新進員工三個月後不會知道當初的 Slack 討論，Agent 也不會知道沒有被寫進 repo 的決策。

OpenAI 在 5 個月內用 Codex 產出 100 萬行程式碼、1,500 個 PR 的實踐中，將此原則發展為核心設計哲學：repo 本身就是唯一的真相來源（single source of truth），所有架構決策、設計文件、產品規格都必須存在 repo 內，而非外部系統。

**在 Claude Code 中的對應**：
- `CLAUDE.md` 作為入口地圖，所有架構決策必須可被 Agent 從 repo 中發現
- 文件結構化存放在 repo 內（`docs/`、`.claude/rules/`），以 `@` 引用連結，而非放在 Notion 或 Google Docs
- 自訂 linter 的錯誤訊息設計為 Agent 可直接執行的修復指令，而非只描述問題

**做對的樣子**：linter 報錯時輸出「此函數缺少型別標註。請在參數和回傳值加上 TypeScript 型別。範例：`function add(a: number, b: number): number`」。Agent 直接讀取錯誤訊息就知道怎麼修。團隊的架構決策記錄在 `docs/design-docs/` 中，CLAUDE.md 用 `@` 指向它。

**做錯的樣子**：linter 報錯只有 `error TS7006: Parameter implicitly has an 'any' type`。Agent 需要額外搜尋文件才能理解修復方式，成功率下降。團隊的架構決策記錄在 Confluence 裡，Agent 完全無法存取。

### 3. 七大原則 × Claude Code 對照總表

| 原則 | Claude Code 機制 | 你在哪章學的 |
|------|-----------------|-------------|
| 上下文架構 | CLAUDE.md + Skills + `@` 匯入 | 02-1, 03-3 |
| 架構約束 | Hooks + deny 規則 | 03-1, 02-2 |
| 自驗證循環 | Stop Hook Validator | 03-2 |
| 上下文隔離 | Subagents + Worktree | 04-1, 04-3 |
| 熵管理 | Auto Memory + 踩坑閉環 + GC Agent | 02-1, 04-2 |
| 可拆卸性 | 模組化檔案結構 | 全書 |
| Agent 可讀性 | CLAUDE.md + 結構化文件 + linter 訊息設計 | 02-1, 03-1 |

這張表的用途：當你在建立或審查 Harness 時，每一列就是一個檢查點。不是「我有沒有用這個機制」，而是「我用這個機制是否正確達成了對應原則的目標」。

## 實作範例：用七大原則審視 AI Dev Assistant

> 主案例延續：在 AI Dev Assistant 專案中，你已完成模組 1-5 的所有設定。現在用七大原則作為檢查框架，審視整個 Harness 是否真的有效，而非只是「存在」。

**目標**：對 AI Dev Assistant 的現有 Harness 逐一套用七大原則，找出缺口並記錄改善項目。

**前置條件**：
- 已完成模組 1-5 的所有實作（CLAUDE.md、Hooks、Skills、Subagents、Worktree）
- 本機的 `ai-dev-assistant` 專案可正常執行 `npm test` 和 `npm run lint`
- Node.js 18.x、TypeScript 5.x、Jest 29.x、ESLint 8.x 已正確安裝

**步驟**：

1. 列出目前 AI Dev Assistant 的所有 Harness 組件，建立清單：
   ```bash
   find .claude -type f | sort
   cat CLAUDE.md | wc -l
   ```
   記錄輸出。這是你的「現況快照」，後續比對用。

2. 建立一份七大原則檢查表（可以是純文字檔案 `harness-audit.md`），每條原則三個狀態：完成 / 部分完成 / 缺失。先不評分，繼續往下檢查後再填。

3. 執行原則 1 檢查——上下文架構是否到位：
   - CLAUDE.md 行數是否低於 200 行？（`wc -l CLAUDE.md`）
   - CLAUDE.md 的內容是索引式的（「深層規範見 `docs/api-rules.md`」），還是百科式的（把所有規範直接貼入）？
   - 有沒有 Skill 覆蓋「按需載入」的知識區塊？
   將結論填入檢查表。

4. 執行原則 2 檢查——架構約束是否使用工具層而非 prompt 層：
   - 列出 CLAUDE.md 中所有「請不要...」「避免...」的條目——這些是 prompt 建議，不是約束
   - 逐一判斷：這條規則如果 Claude 違反了，後果嚴重嗎？如果嚴重，應升級為 Hook
   - 現有的 Hook 是否覆蓋了最高風險的操作？（例如 force push、刪除 production 資料、直接修改 `main` 分支）

5. 執行原則 3 檢查——自驗證循環是否在運作：
   - Stop Validator Hook 是否存在並啟用？（`cat .claude/hooks/stop-validator.sh`）
   - 手動觸發一次：在 Claude Code 中完成任何任務後，觀察是否自動跑 `npm test && npm run lint`
   - 如果測試失敗，Stop Validator 是否真的阻止了 Claude 結束？

6. 執行原則 4 檢查——上下文隔離是否實際被使用：
   - 最近三次複雜任務（超過 10 個檔案、超過 5 輪來回），你是否使用了 Subagent？
   - 如果沒有，主對話的 context 在任務結束後有多大？（從 Claude Code 的 token 計數觀察）
   - Worktree 是否正確設定並在需要時被使用？

7. 執行原則 5 檢查——熵管理是否有閉環機制：
   - 有沒有踩坑日誌？（例如 `.claude/pitfalls.md` 或類似檔案）
   - 上次 review 踩坑日誌是什麼時候？距今多少天？
   - CLAUDE.md 中最老的條目是什麼時候加的？有沒有已不適用的規則？

8. 執行原則 6 檢查——組件是否可以獨立移除：
   - `.claude/hooks/` 目錄中，每個腳本是否可以單獨刪除而不影響其他腳本？
   - 最近 30 天，每個 Hook 有被觸發嗎？（可在腳本中加 log 來追蹤，或觀察 Claude Code 的輸出紀錄）
   - 有沒有某個 Hook 對應的問題，最新版 Claude 已經可以自己處理，不再需要硬性攔截？

**預期結果**：
- `harness-audit.md` 填寫完成，每條原則都有明確的狀態評分（完成 / 部分完成 / 缺失）
- 至少找到 1-2 個具體的改善項目（例如：「原則 2 缺失：CLAUDE.md 有三條『請不要』規則應升級為 Hook」）
- 下一步行動清單：每個「部分完成」或「缺失」對應一個具體的修改計畫

**常見錯誤**：

- **只檢查「有沒有」，不檢查「是否有效」**：Stop Validator 腳本存在，但腳本中有語法錯誤導致它從不正確執行——這等於沒有。Hook 存在，但觸發條件設定錯誤，從未被觸發——這等於沒有。審計必須包含功能驗證，不只是檔案存在確認。

- **不做定期審計，只在出問題時才 review**：Harness 的熵增是緩慢的，單一組件的過時不會立刻造成明顯問題。等到組件互相衝突才發現，修復代價遠大於定期維護。建議每月固定一次 15 分鐘的 Harness 審計，比出問題後花三小時排查便宜。

## 關鍵重點
- Harness Engineering 的核心命題：Agent 的可靠性瓶頸在模型周圍的系統，不在模型本身
- 七大原則中，原則 2（架構約束）最常被忽略——工程師習慣在 CLAUDE.md 寫建議，而非在 Hook 層寫約束
- 原則 3（自驗證循環）是投報率最高的單一機制——Stop Validator 幾乎能在不增加任何對話輪次的情況下，大幅提升交付品質
- 原則 6（可拆卸性）決定了 Harness 的長期維護成本——模組化設計在建立時多花 10% 的時間，但在後續維護上節省 50% 以上的時間
- 七大原則是一個整體：缺少任何一條，其他六條的效益都會打折
