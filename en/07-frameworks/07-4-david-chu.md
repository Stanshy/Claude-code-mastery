# 07-4 David Chu Evolution Framework
> A five-layer evolution model for AI code agents, and the path from graduating CLAUDE.md rules to Hooks, Skills, and Agents.

## Learning Objectives
- Understand what problem each layer of the five-layer evolution model solves, and the trigger conditions for evolving from one layer to the next
- Master the three "graduation" paths for CLAUDE.md rules: the criteria for upgrading to a Hook, Skill, or Agent
- Learn to identify the three graduation signals and know when it is time for rules to evolve
- Understand the differences between the Boris school and the RPI school, and how the integrated perspective unifies the two approaches

## Main Content

### 1. Background: Reflecting on the Journey from High Efficiency to Mental Exhaustion

David Chu is a former software engineer on the ENS team. By surface-level metrics, his workflow was highly successful: running 3 to 5 Claude Code instances simultaneously each day, 100% of code generated by Claude Code, and merging 2 to 4 PRs daily.

But he found himself mentally exhausted at the end of each day.

The cause was not excessive workload, but the nature of the work: he spent a huge amount of time constantly switching terminals, reviewing outputs, and re-prompting. These actions did not require judgment, yet they consumed enormous attention. He felt like he was "managing" Claude rather than "collaborating with" Claude.

This frustration led him to think about a more fundamental question: **Collaboration with Claude Code should not just be a loop of "assign a task, wait for output, give feedback" — it should be a system that evolves over time, requiring less and less human intervention.** This thinking gave birth to the five-layer evolution model.

---

### 2. The Five-Layer Evolution Model

| Layer | Name | Characteristics | What You Are Doing |
|-------|------|-----------------|-------------------|
| Layer 1 | Conversational | Enter prompts, Claude responds | Manually driving every step |
| Layer 2 | Configuration-based | CLAUDE.md guides, memory persists | Rules take effect automatically |
| Layer 3 | Automated | Hooks enforce, Skills encode | Rules are guaranteed to execute |
| Layer 4 | Orchestrated | Custom Agents each handle their own role | Specialized roles handle specialized problems |
| Layer 5 | Autonomous | Agent teams operate independently | You only do the final review |

Each layer's evolution solves a specific problem from the previous layer:

**Layer 1 to Layer 2:** Solves the problem of "having to re-explain the context for every task." CLAUDE.md makes rules persist across sessions — you no longer need to repeat "our code style is like this" every time.

**Layer 2 to Layer 3:** Solves the problem of "Claude understands the rules but does not consistently follow them." Hooks elevate rules from "things Claude should remember" to "things that are technically guaranteed to happen." Skills transform workflows from "describe from scratch every time" to "one command triggers the complete workflow."

**Layer 3 to Layer 4:** Solves the problem of "a single Agent handling all types of tasks leading to decreased focus." When your tasks begin to diverge — some requiring a security perspective, some a performance perspective, some a user experience perspective — specialized Agents are more reliable than generic configurations.

**Layer 4 to Layer 5:** Solves the problem of "humans still needing to coordinate work across multiple Agents." Once collaboration between Agents is systematized, the human role shifts from "coordinator" to "final reviewer."

Evolution is neither mandatory nor linear. You can operate at Layer 2 for a very long time, as long as CLAUDE.md effectively solves your problems. The trigger condition for evolution is always "the current layer cannot reliably solve a certain class of problems."

### 3. Three Graduation Paths

"Graduation" is the metaphor David Chu uses: when a CLAUDE.md rule has matured enough to be taken over by a stronger mechanism, it graduates from CLAUDE.md and gets upgraded to a Hook, Skill, or Agent.

Determining where to upgrade depends on the nature of the problem.

---

#### Path One: CLAUDE.md → Hook

**Applicable scenario:** When Claude no longer consistently follows a certain rule.

Text-based rules have an inherent limitation: during long conversations or complex tasks, Claude's attention may drift away from the rules. If a rule is violated once, twice, three times, that rule is a candidate for a Hook.

**Three key types of Hooks:**

- **PreToolUse**: Intervenes before tool execution. Used for conditional checks — if a certain condition is not met, block tool execution. For example: before any `Write` tool execution, check whether the target path is within an allowed range.
- **PostToolUse**: Intervenes after tool execution. Used for callback validation — if the tool output does not meet expectations, trigger a correction workflow. For example: after `Bash` tool execution, check whether code formatting meets standards, and if not, automatically run the formatter.
- **Stop**: Intervenes when Claude is about to finish a task. Used as a quality gate — confirm all necessary steps have been completed before allowing termination. For example: before Claude finishes, confirm that tests have passed.

**The Math of Validators:**

This is the most important concept for understanding the value of Hooks. Suppose you have a 5-step workflow where Claude's success rate at each step is 80%:

- Without a validator: 0.8 × 0.8 × 0.8 × 0.8 × 0.8 = 33% overall success rate

After adding a validator (assuming 70% validator accuracy), Claude gets a chance to retry failed steps, with up to 4 retries per step:

- With a validator: 1 - (1 - 0.7)^4 × per step ≈ 99% overall success rate

This math illustrates one thing: **A good validator paired with a poor workflow beats a good workflow with no validator.** Validators are the single highest-leverage investment in a Claude Code system.

---

#### Path Two: CLAUDE.md → Skill

**Applicable scenario:** Two situations call for upgrading a rule to a Skill:

1. **Infrequently used workflows**: If a workflow is only used once a month, its rules should not permanently reside in CLAUDE.md taking up context space. Write it as a Skill and load it only when needed.

2. **Knowledge the model was not trained on**: If you are using a specific version of a framework, an internal library, or a niche tech stack, Claude's training data may not contain sufficient relevant knowledge. Encapsulate such knowledge as a Skill and inject it precisely when needed.

**Two concrete examples:**

- **Dexie.js setup Skill**: If you use Dexie.js for a frontend database, Claude may not have sufficient grasp of its best practices. Write "how to properly set up Dexie.js schema, migration, and reactive queries" as a Skill. Before every task involving Dexie.js, execute this Skill first.

- **Team UI preferences Skill**: If your team has specific UI component standards (e.g., all modals must use the `<Dialog>` component, all forms must use `react-hook-form`, all error states must display a toast), write these standards as a Skill and load it before starting frontend tasks.

---

#### Path Three: CLAUDE.md → Agent

**Applicable scenario:** When a class of problems requires sustained expert judgment across multiple tasks.

**The fundamental difference between Skills and Agents:**

- For a Skill, you define a **task**: "Execute this workflow"
- For an Agent, you define a **persona**: "You are a role with this knowledge and judgment"

A Skill is a static process; an Agent is a dynamic role. An Agent can apply its role knowledge across different contexts and make contextual judgments that a Skill cannot.

**Two concrete examples:**

- **security-reviewer Agent**: Not just a security checklist, but a role that continuously evaluates all decisions from a security perspective. It knows your system architecture, understands which parts are high-risk, and provides expert judgment in any task involving authentication, authorization, or data handling.

- **debugger Agent**: Not just a "find bugs Skill," but a role familiar with your codebase, aware of common problem patterns, and capable of quickly narrowing down root causes based on symptoms. It remembers similar issues fixed last week and can connect patterns across time.

### 4. Three Graduation Signals

David Chu identified three signals in practice. When any of them appears, it is time to review your CLAUDE.md and consider graduation:

**Signal One: CLAUDE.md reaches approximately 50 lines**

When CLAUDE.md exceeds 50 lines, you need to start asking: which rules are needed every day? Which are only used in specific contexts? The latter are candidates for Skills.

50 lines is not an absolute threshold but a review checkpoint. A CLAUDE.md of this length may already contain enough "context-specific rules" to warrant a round of cleanup.

**Signal Two: Claude no longer consistently follows a certain rule**

This is the clearest signal. If you find yourself repeatedly reminding Claude of the same thing across multiple sessions, that rule needs to be upgraded from the text level to the technical level — it needs to become a Hook.

**Signal Three: It is time for a scheduled review**

David Chu recommends setting a fixed CLAUDE.md review cadence, such as every two weeks. The purpose of this periodic review is to proactively identify which rules have "matured" enough to graduate, rather than waiting for problems to occur and reacting passively.

### 5. Integrating the Boris School vs. the RPI School

On the question of how "rich" CLAUDE.md should be, there are two opposing views in the community:

| School | Viewpoint | Applicable Stage |
|--------|-----------|-----------------|
| Boris Cherny | Invest relentlessly in CLAUDE.md — add a rule for every mistake | Early stage, building habits |
| RPI School | Keep CLAUDE.md under 60 lines; exceeding 40% of context enters the "dumb zone" | Mature stage, streamlining |
| Integrated view (David Chu) | Start with Boris's method to accumulate, then graduate rules once they mature | Entire lifecycle |

**Core claim of the Boris school:** Every time Claude makes a mistake, add a rule to CLAUDE.md. Do not hold back, do not hesitate. This is the fastest way to build institutional memory.

**Core claim of the RPI school:** An overly long CLAUDE.md has adverse effects. When CLAUDE.md occupies more than 40% of the context window, Claude's overall performance degrades — it enters the so-called "dumb zone," where all capabilities are affected. CLAUDE.md needs to be streamlined, keeping only the most essential rules.

These two views appear contradictory, but David Chu's integrated perspective explains that they are actually talking about different things:

Boris is talking about the **early stage**: accumulate quickly, build the habit of turning mistakes into rules.

RPI is talking about the **mature stage**: when there are enough rules, periodically streamline by graduating mature rules out, rather than letting them permanently occupy context.

**The secret weapon that makes Boris's dense CLAUDE.md work:**

The reason Boris's CLAUDE.md still works effectively at 2,500 tokens is that he has a validator that enforces CLAUDE.md itself. This validator ensures Claude actually reads and applies CLAUDE.md rules at the start of every task, rather than formally loading them and then ignoring them. This validator has now become part of the `/simplify` command.

**Practicing the integrated approach:**

1. Use Boris's method to accumulate rules (add a rule for every mistake)
2. Periodically review and upgrade stable, high-frequency rules to Hooks
3. Extract context-specific rules into Skills
4. Upgrade roles requiring sustained expert judgment to Agents
5. Keep CLAUDE.md as a streamlined collection of "living rules"

---

### 6. The Direction of System Growth

David Chu's framework ultimately provides a very clear direction:

- **When Claude makes a mistake, first add the rule to CLAUDE.md** — let the error happen only once.
- **When Claude no longer follows a rule, upgrade it to a Hook, Skill, or Agent** — elevate the rule from text to mechanism.
- **This ensures your tooling grows from actual friction encountered, rather than premature optimization** — every evolution of the system is driven by real needs, not theoretical design.

This is an antifragile approach to system design: every failure makes the system stronger, rather than requiring manual repair.

## Teaching Example

### Scenario: The Complete Lifecycle of a CLAUDE.md Rule from Creation to Graduation as a Hook

**Starting point: The first pitfall (Day 0)**

You are developing a Node.js API with Claude Code. One day, while handling a complex database query task, Claude directly concatenates a SQL query string:

```javascript
// Code generated by Claude
const query = `SELECT * FROM users WHERE id = ${userId}`;
db.query(query);
```

This is a SQL injection vulnerability. You catch it and fix it during code review.

**Step one: Add to CLAUDE.md (Day 0, 2 minutes)**

After the fix, you immediately add a rule to CLAUDE.md:

```
## Security Rules
- All database queries must use parameterized queries
  - PostgreSQL: use $1, $2 placeholders
  - MySQL: use ? placeholders
  - String concatenation for SQL is strictly forbidden
```

Over the next two weeks, this rule works effectively. Claude uses parameterized queries when generating database queries.

**Signal appears: The rule starts being violated (Day 15)**

Your CLAUDE.md has grown to 45 lines and contains multiple rules. During a particularly complex task — involving multi-table joins and dynamic filter conditions — Claude once again generates a string-concatenated query.

You fix it and remind Claude again, but you realize this rule is entering the "no longer consistently followed" state. This is the second graduation signal.

You decide to let this rule graduate.

**Step two: Design the Hook (Day 15, 30 minutes)**

You create a PostToolUse Hook that triggers after every `Write` tool execution (i.e., when Claude writes to a file):

```typescript
// .claude/hooks/sql-injection-guard.ts
export async function onPostToolUse(context: ToolUseContext) {
  if (context.tool.name !== 'Write') return;

  const content = context.tool.output.content as string;

  // Detect patterns of string-concatenated SQL
  const sqlInjectionPatterns = [
    /`[^`]*SELECT[^`]*\$\{/i,           // SELECT in template literals
    /['"][^'"]*SELECT[^'"]*' \+/i,       // SELECT with string concatenation
    /query\s*=\s*['"`][^'"`]*\+\s*\w/i, // query = "..." + variable
  ];

  const violations = sqlInjectionPatterns
    .filter(pattern => pattern.test(content))
    .length;

  if (violations > 0) {
    return {
      action: 'block',
      message: `Potential SQL injection pattern detected.

You must use parameterized queries:
- PostgreSQL: db.query('SELECT * FROM users WHERE id = $1', [userId])
- MySQL: db.query('SELECT * FROM users WHERE id = ?', [userId])

Please fix and resubmit.`
    };
  }
}
```

**Step three: Test the Hook (Day 15, 15 minutes)**

You write a test task that intentionally makes Claude generate a string-concatenated query to confirm the Hook intercepts it and provides the correct guidance. You also test a correct parameterized query to confirm the Hook does not produce a false positive.

Both tests pass. The Hook goes live.

**Step four: Update CLAUDE.md (Day 15, 5 minutes)**

You update the security rule in CLAUDE.md to:

```
## Security Rules
- SQL injection protection is automatically enforced by a PostToolUse Hook (sql-injection-guard.ts)
  - String-concatenated SQL is automatically intercepted; no manual reminders needed
```

The purpose of this note is to let your future self (or anyone reading CLAUDE.md) know: this rule has graduated. There is no need to maintain a detailed description in CLAUDE.md because enforcement is now guaranteed by the Hook.

CLAUDE.md goes from 45 lines down to 42 lines.

**Result: The system is now stronger than before**

- This rule has been upgraded from "something Claude should remember" to "something technically guaranteed to happen"
- Even if Claude's attention drifts during complex tasks, the Hook still guarantees rule enforcement
- CLAUDE.md did not bloat because of this rule — it actually became leaner through graduation

**Review 15 days later (Day 30)**

You perform your periodic CLAUDE.md review. The Hook has executed 23 times, intercepting 4 violations. All 4 occurred during complex multi-table join tasks — exactly the scenarios where rules are most easily forgotten in long contexts.

You confirm the Hook design is correct. You also notice the Hook once falsely intercepted a template string that was not SQL — you update the pattern rules to make them more precise.

This is the complete evolution lifecycle of a rule: from pitfall → CLAUDE.md rule → Hook. Every step is driven by real needs, and every step makes the system more reliable.

## Key Takeaways
- Every upgrade in the five-layer evolution model is driven by a concrete problem, not theoretical design; the trigger condition for evolution is "the current layer cannot reliably solve a certain class of problems"
- The combination of a PostToolUse Hook with validation logic can boost overall success rates from 33% to nearly 99%, making it the single highest-leverage investment in the system
- The fundamental difference between Skills and Agents: for a Skill you define a task, for an Agent you define a persona
- The Boris school (rapid accumulation) and the RPI school (streamlined context) are not contradictory: the former applies to the early stage of building habits, the latter to the mature stage of streamlining — the graduation mechanism connects the two
- Systems should grow from actual friction encountered: first add to CLAUDE.md, then upgrade to Hook/Skill/Agent when rules are no longer followed — avoid premature optimization
