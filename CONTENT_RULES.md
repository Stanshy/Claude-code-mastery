# 內容生成規範

## 硬規則（違反則重寫）

1. **技術內容 100% 精確**：不確定的必須標註「需驗證」
2. **不允許空泛描述**：禁止「提升效率」「很方便」「非常強大」
3. **一章一能力**：每章只教一個核心能力，不混雜
4. **範例可執行**：所有程式碼和配置必須能實際跑
5. **主案例延續**：每章的範例都在 AI Dev Assistant 上累加，技術棧與 PROJECT_SPEC.md 一致

---

## 實作範例強制 5 元素

每個實作範例必須包含：

| 元素 | 說明 | 範例 |
|------|------|------|
| **目標** | 這個範例要達成什麼（一句話） | 「為 AI Dev Assistant 加入 ESLint 自動格式化 Hook」 |
| **前置條件** | 需要先完成什麼（列出章節或步驟） | 「已完成 01-2（專案骨架建立）」 |
| **步驟** | 逐步操作（不省略關鍵步驟） | 1. 建立檔案... 2. 編輯設定... |
| **預期結果** | 成功後會看到什麼（含具體輸出） | 「終端顯示 ✓ ESLint passed」 |
| **常見錯誤** | 至少 2 個（錯誤 → 原因 → 解法） | 「Error: Cannot find module → 未安裝依賴 → npm install」 |

---

## 正文寫作規則

- 先說「為什麼」再說「怎麼做」
- 用工程師視角解釋，不用行銷語言
- 每個概念用「定義 → 原理 → 範例 → 常見誤解」結構
- 對比時用表格，不用長段落
- 進階技巧標註為「進階」，不混入基礎內容

---

## 禁止事項

- 省略關鍵步驟的範例
- 無法實際執行的假程式碼
- 與其他章節重複的內容
- 未經確認的版本號或 API
- 沒有上下文的程式碼片段

---

## 技術驗證（工程層）

每章生成後，檢查以下項目：
- CLI 指令是否真實存在（不可有虛構的 flag 或 subcommand）
- npm / Node 指令是否符合 Node 18.x 語法
- TypeScript import 語法是否正確（ES module style）
- Jest 測試語法是否與 v29 相容
- 配置檔格式是否與官方文件一致（settings.json / .mcp.json / SKILL.md frontmatter）

## 閱讀流暢度驗證

每章生成後，確認以下三點：
1. 第一段是否回答「為什麼要學這個能力」
2. 是否在 2-3 分鐘閱讀量內看到第一個可執行步驟
3. 實作範例是否在 6-10 步內完成

---

## 失敗重試機制

若內容不符合規範：

1. 列出違規項目（具體指出哪條規則、哪個段落）
2. 只重寫違規段落（不整章重寫）
3. 最多重試 3 次
4. 若仍不合格，標註「需人工檢查」並說明卡點
